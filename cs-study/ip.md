# 공인 IP / 사설 IP / NAT / 포트포워딩 정리

## 1. 공인 IP와 사설 IP가 나뉘는 이유

### 1-1. IPv4 주소 고갈 문제
- IPv4는 주소 개수가 **약 43억 개**로 한정되어 있음
- 전 세계 인터넷 연결 기기 수(컴퓨터, 폰, IoT 기기 등)는 이보다 훨씬 많음
- 모든 기기에 공인 IP를 하나씩 부여하면 주소가 절대적으로 부족해짐

### 1-2. 해결책 — 공인 IP / 사설 IP 이원화
- **공인 IP**: 전 세계에서 유일하게 할당되는 주소. 인터넷 상의 라우터들이 이 주소로 가는 길을 정확히 알고 있음
- **사설 IP**: 각 로컬 네트워크(집, 회사 등) 내부에서만 유효한 주소 (`192.168.x.x`, `10.x.x.x` 등). 전 세계 여러 곳에서 동일한 사설 IP가 중복되어 쓰임
- 인터넷 라우터들은 사설 IP 대역으로는 아예 길 안내(라우팅)를 하지 않도록 표준(RFC 1918)으로 정해져 있음 → 그래서 외부에서 사설 IP로 직접 접근 자체가 불가능함

### 1-3. 가정에서의 실제 구조
```
통신사 회선 (세대별로 개별 계약)
   ↓
공유기 한 대 연결
   ↓
공유기가 공인 IP 하나를 통신사로부터 할당받음
   ↓
공유기에 연결된 내부 기기들(컴퓨터, 폰 등)은
공유기가 부여하는 사설 IP를 각각 받음
```
- 공인 IP를 갖는 단위는 **"세대(계약) = 공유기 한 대"**
- 아파트 전체가 아니라 **각 호실(세대)마다 별도로 공인 IP 하나씩** 할당받음
- 세대 내 벽에 있는 여러 개의 랜선 단자(방마다 있는 것)는 대부분 **세대 내 통신단자함에서 나온 배선이 여러 갈래로 분배**된 것뿐이며, 실제 통신 회선(계약)은 하나인 경우가 대부분
  - 이 경우 어느 단자에 공유기를 연결하든 **최초로 연결·인증된 공유기 하나만 공인 IP를 받음**
  - 여러 단자에 공유기를 동시에 여러 대 연결해도 공인 IP가 여러 개로 늘어나지 않음 (통신사 계약이 회선 1개 기준이기 때문)
  - 단, 세대에 통신 회선 자체가 여러 개 별도로 계약되어 있다면(드문 경우) 각각 독립된 공인 IP를 받을 수 있음
- 벽에 있는 랜선 단자 자체는 공유기를 꽂기 전까지는 공인/사설 개념이 없는, 그냥 신호가 흐르는 물리적 통로일 뿐임

## 2. NAT (Network Address Translation, 네트워크 주소 변환)

공유기가 사설 IP ↔ 공인 IP를 서로 변환해주는 기술.

### 2-1. 내부 → 외부 (요청을 보낼 때)
```
1. 내 컴퓨터(사설 IP)가 외부로 요청을 보냄
2. 공유기가 이 요청의 출발지 주소를
   사설 IP → 공인 IP로 바꿔서 내보냄
3. 이때 공유기는 "이 요청은 어떤 사설 IP가 보낸 것"인지
   내부 표(NAT 테이블)에 기록해 둠
4. 외부 서버가 응답을 공인 IP로 돌려보내면,
   공유기가 NAT 테이블을 보고 원래 요청했던
   사설 IP로 다시 전달해 줌
```
- 이 과정은 **자동으로** 이루어짐

### 2-2. 외부 → 내부 (외부가 먼저 접속을 시도할 때)
```
1. 외부에서 공인 IP로 갑자기 접속을 시도
2. 공유기 입장에서는 "이런 요청을 보낸 적도 없고,
   NAT 테이블에 기록된 것도 없음"
3. 어느 내부 기기로 전달해야 할지 판단 근거가 없어
   기본적으로 요청을 차단/거부함
```
- 이게 사설 IP로 외부 접속이 원천적으로 안 되는 이유

## 3. 포트포워딩 (Port Forwarding)

- 위 2-2번 상황(외부 → 내부)을 가능하게 만들기 위해, **사람이 공유기에 직접 규칙을 미리 등록**해두는 설정
- NAT처럼 자동으로 동작하는 게 아니라 **수동으로 설정**해야 함

```
설정 예시:
"공인 IP의 3000번 포트로 들어오는 요청은
 무조건 내부 192.168.0.5:3000 (내 서버)으로 전달해라"
```

```
외부 요청 → 공인 IP:3000 도착
   ↓
공유기: 포트포워딩 규칙 확인
   ↓
내부 사설 IP:3000 (내 Node 서버)로 전달
   ↓
서버가 응답 → 역순으로 다시 외부로 전송
```

## 4. NAT와 포트포워딩의 관계 정리

| 구분 | NAT (일반) | 포트포워딩 |
|---|---|---|
| 방향 | 내부 → 외부 (요청을 내가 먼저 보낼 때) | 외부 → 내부 (외부가 먼저 접속을 시도할 때) |
| 동작 방식 | 자동 | 수동 설정 필요 |
| 목적 | 사설 IP를 가진 기기들이 공인 IP로 인터넷에 나가게 해줌 | 외부에서 내가 운영하는 서버(내부 기기)에 접근할 수 있게 열어줌 |
| 관계 | 더 큰 개념 (주소 변환 기술 전체) | NAT 기술 중, 외부 요청을 받아주기 위한 특수 설정 |

## 5. 실무에서의 참고

- 포트포워딩은 "집 컴퓨터를 직접 서버로 쓰고 싶을 때" 필요한 개념
- 실무에서는 아래 방식이 더 많이 쓰여서 포트포워딩을 직접 설정할 일이 상대적으로 적음
  - **클라우드 서버**(AWS EC2, GCP 등): 서버 자체가 처음부터 공인 IP를 가짐 (대신 보안그룹/방화벽 설정을 함)
  - **배포 플랫폼**(Vercel, Railway, Render 등): 포트포워딩 신경 쓸 필요 없이 자동으로 외부 노출됨
- 다만 원리를 알아두면 "왜 배포가 이런 방식으로 동작하는지" 이해하는 데 도움이 됨
